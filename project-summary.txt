2. Database Schema & Multi-Tenant Design
I initialized Prisma (the new Prisma 7 convention with prisma.config.ts) and configured the relational schema for our multi-tenant SaaS.

I added the following core tables to prisma/schema.prisma (you can open this file to review the structure):

User, Account, Session, VerificationToken: This is the standard schema required by NextAuth.js (which we will configure next). It safely stores users and maps them to external OAuth providers (like Google).
Workspace: This is the core of our Multi-Tenancy. An organization is represented by a workspace. We also included nullable Stripe fields (stripeCustomerId, stripeSubscriptionId, etc.) here. Why? Because in a B2B SaaS, the organization pays for the subscription, not individual users.
WorkspaceMember: This is a pivot (join) table connecting User and Workspace with a Role enum (e.g. ADMIN or MEMBER). This allows exactly what you asked for: Role-Based Access Control (RBAC) within the context of a tenant. A single user can be an Admin of "Workspace A", but merely a Member of "Workspace B".
TIP

Senior Polish: Notice the @@unique([userId, workspaceId]) index on WorkspaceMember. This prevents the same user from being accidentally added to the same workspace more than once! Also, notice we use onDelete: Cascade on the Account and WorkspaceMember relations. This prevents orphaned rows—if a user deletes their account, all their memberships are instantly wiped.


What You Just Learned:
The Shared Schema Model: By giving every piece of tenant data a workspaceId link, we achieve data isolation efficiently.
Join Tables for RBAC: Direct User-to-Workspace links are insufficient. The WorkspaceMember table allows us to store metadata (like role) for that specific membership. B2B billing attaches to the Workspace, not the individual.

Here is a breakdown of the code we just wrote and why these architectural decisions are so powerful in modern Next.js 15:

1. The Controller Page (/src/app/(dashboard)/page.tsx)
When a user hits /dashboard, this React Server Component (RSC) instantly queries the database to grab their WorkspaceMember row. If we find one, we immediately redirect them to /dashboard/their-workspace-slug. If we don't, we bounce them to /onboarding. Why this is good: Because this runs on the server, the user never sees a "loading spinner" or a flash of an empty screen before the redirect happens.

2. The Form without JavaScript! (/src/app/onboarding/page.tsx & workspace.ts)
We created a beautiful, simple UI for the onboarding form and attached it to a Next.js Server Action (createWorkspaceAction). When the user clicks "Create Workspace":

The server generates a clean URL slug (e.g. My Team -> my-team).
We use a Prisma Transaction (prisma.$transaction) to guarantee that the Workspace and WorkspaceMember are created at the exact same moment. If one fails, the entire action rolls back safely.
The server instantly redirects them to their new dashboard URL.
3. The Titanium Security Layout (/src/app/(dashboard)/[workspaceSlug]/layout.tsx)
This is the core of Phase 4 and Phase 5 combined. Any page inside this folder inherits this layout.

The Check: We run a Prisma query against WorkspaceMember searching for both the logged-in User ID and the current URL's workspaceSlug.
The Response: If Prisma returns null (meaning this user has no role in this workspace), we execute notFound(), which safely throws a clean 404 page rather than exposing private data.
The Layout: If they pass the check, we render a beautiful dark-mode sidebar with dynamic routing links!

What We Built & The Architecture
The Feature Gating Model (hasActiveSubscription): Look at src/lib/stripe.ts. This single, powerful function compares the workspace.stripeCurrentPeriodEnd stored in your PostgreSQL database with the exact current timestamp. It adds a 1-day grace period to prevent Stripe webhook delays from instantly locking a customer out of their account.

The Billing UI & Conditional UI Trees: Inside /billing/page.tsx, we have a beautiful pricing table. Notice how it structurally checks if isPro is true. If they are already Pro, the UI gracefully transforms down into a "Manage Subscription" state. It also strictly checks the RBAC membership.role. If you downgrade someone from ADMIN to MEMBER, the Upgrade button immediately disables itself for them.

Secure Checkout (Server Actions + Stripe Client): When an Admin clicks "Upgrade", it fires createCheckoutSession in src/app/actions/billing.ts.

It checks their exact workspaceId.
It creates a real Customer object in Stripe if this is their first time clicking the button.
It embeds metadata: { workspaceId } immediately into Stripe. This is the most critical step of a SaaS application, because when Stripe charges their card and sends a Webhook notification back to our app, we use that metadata ID to link the payment back to the correct database row!
The Webhook Endpoint: I created src/app/api/stripe/webhook/route.ts. This acts as your "cash register receiver". When Stripe completes a payment, it hits this endpoint securely, verifying its cryptographic signature, and updates your Prisma stripeCurrentPeriodEnd fields instantly.

Because you are running on localhost, Stripe cannot natively "talk" to your computer. Stripe built a CLI to securely tunnel these webhooks directly to your local development server!
